# Vertex CRM & Recruitment Operations Platform

## Phase workflow (mandatory)

This platform is being built phase-by-phase against an approved SRS. For **every** phase:

1. **Before starting**, re-read all three source documents:
   - `docs/PLATFORM_ARCHITECTURE.md` — the architecture and phased delivery plan for this build
   - `/Users/dunc/Desktop/Vertex/Vertex/Vertex_SRS_Platform_Phases_1-5.docx` — the approved SRS (functional/non-functional requirements, phase exit criteria)
   - `/Users/dunc/Desktop/Vertex/Vertex/Vertex platform recommendation.pdf` — the approved build/buy/integrate decision and gap analysis
   (The SRS is a `.docx` — the Read tool can't parse it directly; extract text via a `zipfile`/regex script against `word/document.xml`, as done when the architecture doc was first written.)
2. **Implement** the phase's scope as scoped in `docs/PLATFORM_ARCHITECTURE.md` — don't silently expand or shrink scope from what that doc and the SRS's per-phase Functional Requirements define.
3. **Test** the phase's changes (build passes, relevant unit/E2E coverage, manual verification of the new flows).
4. **Verify against the exit criteria** in the SRS (§8, Phasing & Acceptance) and the architecture doc's per-phase "Exit" line before considering the phase done or moving to the next one.

Do not start the next phase until the current one's exit criteria are met.

## Delivery approach

Sequential, phase-by-phase — not parallelized across phases (confirmed with the business).

## Git / environment workflow

- `main` — production.
- `staging` — pre-production sign-off environment. Every phase branch (`phase-N-*`) is merged into `staging` first; a phase is only merged from `staging` into `main` after it's been verified there against the SRS exit criteria and signed off.
- `phase-N-*` — one branch per phase, branched off the **latest `staging`**, not `main` — `main` only updates after formal business sign-off, which lags behind engineering completion, so it's frequently behind `staging` by one or more completed phases. Branching off `main` would silently drop whatever's already in `staging`.
- **On completing a phase**: push the `phase-N-*` branch, then merge it into `staging` **locally** (`git checkout staging && git merge phase-N-* --no-edit`) and push `staging` — do this automatically for every completed phase without being asked. Don't merge to `main` without explicit confirmation.
- Action item (not doable from the CLI): confirm the cPanel Node hosting is set up to run separate `staging` and `main` deployments (matching the SRS §2.3 environment requirement), not just a single production instance.

## Notable bug found and fixed during Phase 1 (not something Phase 1 introduced)

`middleware.ts` used `jsonwebtoken`'s `jwt.verify`, which depends on Node's `crypto` module — **not supported in the Edge runtime that Next.js middleware defaults to**. This meant `verifyToken()` silently threw on every request, so **the entire auth gate always redirected back to login even with a fully valid session cookie** — this predates Phase 1 entirely (the original binary candidate/admin middleware had the exact same bug) and was probably never caught because no one had tested a real login against a live database before. Fixed by adding `runtime: "nodejs"` to `middleware.ts`'s exported `config`.

**Migrated to `proxy.ts` during Phase 3** (2026-07-08), closing the "re-verify this still holds" note above. `src/middleware.ts` → `src/proxy.ts`, `export function middleware` → `export function proxy`, and — this is the part that would have silently reintroduced the exact Edge-runtime bug if missed — **the `runtime: "nodejs"` config key was deleted, not kept**: Proxy files don't accept a `runtime` option at all (throws a build error if present) and default to the Node.js runtime unconditionally as of Next.js 16. Verified by rerunning `e2e/rbac.spec.ts`'s "regression coverage for the middleware crypto bug" test (still passes) and a live `curl` against a protected route confirming the 307-to-login redirect still fires.

## Bugs found and fixed during Phase 2

- `POST /api/upload` looked up the candidate by `user_id: user!.userId` unconditionally — meaning only a self-registered candidate logged into their own account could ever upload a document. A recruiter-sourced lead has no account, so the screening gate's documentation check was unreachable for the entire agent-network pathway. Fixed by accepting an optional `candidate_id` query param, gated by `isStaffRole` + `canAccessCandidate`, so a recruiter/supervisor can upload on a lead's behalf.
- `CandidateStatusControls` showed an "Advance to X" / "Verify → X" button purely based on `STATUS_ORDER` position, regardless of whether the viewer's role was actually allowed to make that specific move (e.g. a recruiter past "reported," or a supervisor before it). The API correctly rejected these (no security issue), but the UI looked actionable and always failed — fixed by gating the buttons on the same reported-boundary logic `candidateLifecycle.ts` already enforces server-side, with an "Awaiting supervisor action" / "Awaiting recruiter to report" message in the dead zones instead.
- `z.string().email().optional()` in Zod rejects an untouched form field sending `""`, since `.optional()` only short-circuits `undefined`, not an empty string — a genuinely optional email field was practically mandatory. Fixed with a `z.preprocess` that maps `""` to `undefined` first (`optionalEmail` in `validations.ts`).

## Bugs found and fixed during Phase 4

- `/admin/applications` (list view) rendered `app.candidate.user.full_name` with no null check — a real recruiter-sourced candidate with no linked account yet (SRS FR-2.1's own documented state) crashes the **entire page** client-side, not just that one row, taking down admin's ability to approve *any* application until fixed. The API already selected the fallback `candidate.full_name`; only the render was missing the `?? candidate.full_name` fallback. Same bug, same fix, in the detail view (`/admin/applications/[id]`) — plus that route's PUT handler would have thrown server-side (500) trying to email a null `user.email` when changing status on such an application, now guarded with `if (existingApp.candidate.user)`.
- The detail view's PUT handler (`/api/admin/applications/[id]`) was a second, independent path to approve an application (the list view's inline dropdown being the first) and was missing the case-auto-creation hook entirely — approving from there silently opened no Case. Both paths now call the same `createCaseForApprovedApplication`.
- `useEffect(load, [])` where `load` is an `async` function fails TypeScript's `EffectCallback` type (a Promise isn't a valid effect return) — caught by the build, not by a passing-then-silently-wrong test. Wrap it: `useEffect(() => { load(); }, [])`.
- `/api/cases` (list endpoint) intentionally returns only `contract: { status }` for the staff case-list table, not full content — `CandidateCaseCard` on the candidate dashboard needs the full contract text to display and sign, and fetching from the list endpoint silently rendered nothing. Fixed by having the candidate-facing component fetch full detail per case (`/api/cases/:id`) — cheap at the scale of "a candidate has 0–1 cases," not worth widening the shared list payload for.
- `RetentionFollowUp` was modeled with `case_id` as its own primary key (`@id`, no separate `id` field), copied verbatim from the architecture doc's original sketch. The audit-log extension's `entityId` extraction always reads `result.id`/`before.id` — every audit entry for this model would have logged `entity_id: "unknown"`, silently breaking traceability for the one model tracking a compliance-relevant 90-day follow-up. Fixed before ever writing a row: gave it a normal generated `id` PK plus `case_id String @unique`, matching every other audited model in the schema.

- A CSS `capitalize` class makes text *display* capitalized without changing the actual DOM text content — bit both the app (no functional impact, purely cosmetic) and this session's own E2E tests twice: `getByText("Weekly Report", { exact: true })` failed because the real text node is `"weekly report"`. ## Concurrent-session collision and isolation fix (2026-07-11)

Mid-session on the Regional Recruiter review phase, a second, unrelated Claude Code session was started in the same working directory (an "Opportunities Catalog" / Job-pricing feature, on `feature/opportunities-catalog`) without either session working in an isolated git worktree. Since a background session edits its working directory in place rather than in an isolated worktree by default, both sessions shared the literal same checkout — the other session's `git checkout`/`git reset` sequence (branching off `staging` after forgetting to do so up front) silently carried this session's checkout along with it, losing an uncommitted new file (`CandidateDetail.tsx`) in the process and leaving both sessions' uncommitted edits intermingled in one working tree.

**Fix, and the pattern to reuse if this happens again:**
1. Identify which files belong to which session (`git status`, cross-referenced against what you've actually edited this session — don't guess).
2. `git add` only your own files by explicit path (never `-A`) and commit them to your own branch, leaving the other session's files unstaged and untouched.
3. `git checkout` the main working directory back to the *other* session's branch, so it isn't left stranded on the wrong branch when it resumes.
4. `git worktree add <path> <your-branch>` + `EnterWorktree` (path form) to move your own session into an isolated checkout going forward — worktrees share the same `.git`/commit history but have independent working directories, so simultaneous sessions stop colliding on `git checkout`.
5. A fresh worktree has no `node_modules`, `.env`, or `.env.test` — copy the env files from the main checkout, and run a real `npm install` rather than symlinking `node_modules`: Turbopack refuses to run against a symlinked `node_modules` ("Symlink node_modules is invalid, it points out of the filesystem root").
6. **Database isolation matters too, not just files.** Both worktrees' `.env` point at the same `vertex_dev`/`vertex_test` by default. Before running `prisma db push` from the new worktree, check whether the *other* session has already pushed schema changes of its own to the shared database (`psql ... \d "TableName"`, or just ask) — `db push` computes its diff against the live database, not against the other session's `schema.prisma`, so it will happily `DROP` columns/tables the other session already added if your own `schema.prisma` doesn't know about them. Safer default once a schema change is needed on either side: create a dedicated scratch database (`createdb vertex_dev_<branch-name>`) and repoint just that worktree's `.env`/`.env.test` at it, `db push` + `db:seed` + rerun `audit-log-immutability.sql` there. Costs a few extra minutes of setup, avoids risking the shared database entirely.
7. Background `nohup ... &` + `disown` processes (e.g. a manually-backgrounded `next dev`) were unreliable in this environment — they died silently between tool calls for no visible reason (not an OOM/crash in the process's own log). The Bash tool's own `run_in_background: true` parameter is what actually stayed alive reliably; prefer it over manual shell backgrounding for anything that needs to keep running across multiple tool calls (a dev server, in particular).
8. Remember to point `playwright.config.ts`'s `baseURL`/`webServer` at your worktree's own port while testing there, and `git checkout -- playwright.config.ts` to revert before committing — the merged code should keep the shared team default (port 3000).
9. **Action item, not yet done:** the shared `vertex_dev`/`vertex_test` databases still need `prisma db push` (adds `RecruiterTarget`) + `npm run db:seed` run against them before this phase's work is testable outside the scratch database it was built and verified against — deliberately left undone to avoid touching a database the other concurrent session was actively using. Do this once that session's own schema work is either merged or otherwise reconciled.

## Key resolved decisions (do not re-litigate without cause)

- Candidate-fee policy (SRS FR-4.5): admin-configurable `FeePolicy` model, defaults disabled — not a leadership-blocking dependency.
- No live candidate data or hardcoded public storage URLs exist in production — schema/storage changes in Phase 1 don't need backfill migrations or call-site audits.
- No API versioning (`/api/v1/`) until/unless external partner API access is actually introduced — Phase 5 shipped a staff-managed Partner CRM with no such access, so this stays deferred.
- Dashboards and new portals reuse the existing design tokens/utility classes in `globals.css` (midnight/gold/ivory palette, Outfit font, Phosphor icons) inside a shared `PortalShell` — no new visual language for internal pages.
- Automated reminders for overdue reports (FR-3.4) are deliberately deferred to Phase 4/5, confirmed with the business 2026-07-08 — not literally required by the SRS's FR-3.4 wording, and needs a background scheduler + due-date concept that don't exist yet. Don't build ad hoc.
- PDF export (FR-3.8, a Should) is deliberately skipped — CSV export (`/api/candidates/export`) substantially satisfies it without adding a new PDF-generation dependency, confirmed with the business 2026-07-08.
- Contract e-signature (FR-4.3) is a real, in-house "simple electronic signature" (typed full-legal-name attestation + authenticated session + timestamp + IP) — deliberately *not* a DocuSign/Dropbox Sign integration. The platform recommendation places that vendor under "Integrate," scoped to Phase 5; this Contract model is the groundwork a vendor swap sits behind.
- Milestone payments (FR-4.5) are a staff-entered *record* of a payment collected (with a receipt reference), gated by `FeePolicy`, defaults disabled — not a new Stripe/PayPal charge flow. That's the separate, already-built candidate application-fee path (`Payment` model).
- Automated (email/push) reminders for permit/visa deadlines (FR-4.4) and stage-based staff notifications (FR-4.8) are deferred, same reasoning as Phase 3's report reminders — deadlines are real staff-entered data with real in-app urgency badges, just not a background scheduler yet.
- `Country` (the existing region-scoped reference-data model, originally built for recruiter/supervisor territory assignment) deliberately doubles as the candidate's destination-programme picker on the Candidate Information Form, scoped to whichever countries admin places under the "Europe" region — no second model was introduced. Source-market regions (Africa, Middle East & Gulf) stay exactly what they were.
- `/apply/hungary` and `/apply/greece` were retired (2026-07-09), confirmed with the business — they were static, pre-pivot mockups with no real submission path (a JS `alert()`, no API call), superseded entirely by the one generic multi-country apply form.
- The Candidate Information Form is filled in *before* any account exists, for both self-service and recruiter-assisted candidates — not after, as first built. Screening gates on the form's own answers; account creation (via emailed invite) and document upload both happen only after screening passes. Confirmed with the business 2026-07-09; see the architecture doc's "Pre-application lifecycle re-sequenced" entry for the full before/after.
- E2E suite stays Playwright (assertions, retries, HTML/trace reports, parallel test files) — `vercel-labs/agent-browser` (a CLI with no test-runner or assertions) is for manual/ad-hoc browser exploration only, not a suite replacement. Confirmed with the business 2026-07-09 after comparing the two.
- The real Vertex logo (a silver/grey stylised "V"/checkmark mark, `public/vertex-logo.png`) replaced the hand-drawn SVG approximation everywhere via a single shared `/logo.svg` (the real PNG embedded as a data URI) — every existing `brightness-0`/`invert` CSS filter usage picks it up automatically. The old SVG had its "VERTEX" wordmark baked into the image itself; the real logo is icon-only, so login/register/the footer needed an explicit "VERTEX" text span added (Navbar/PortalShell already had one, just not those three).
- FR-5.3–5.9 (M365/Teams, Calendly, e-signature vendor, BI layer, WhatsApp Business, partner-acquisition tooling) are confirmed-deferred, not built as stub/adapter code — no third-party accounts exist yet, and unused adapter interfaces against an unchosen vendor would be speculative work. Confirmed with the business 2026-07-10: build the core CRM now, revisit integrations once real accounts exist.
- `computeRecruiterResponseRate` (KPI) keeps its existing disclosed interpretation (recruiter follow-through rate) even though `Partner` now exists — it was already honestly labeled "Recruiter Response Rate" in the UI and well-tested, so it wasn't repurposed. Don't add new inline-editing affordances to the candidate list row going forward; extend the detail page instead.
